# Story 10.5: Persist a moved comment box's position

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a reader,
I want a comment box I dragged to a new spot to stay there,
so that I can place a note where it reads best and keep it there.

## Acceptance Criteria

1. **(Move persists for the session, item 5, FR-11/FR-15)** Given an open comment box (bubble), when I drag it to a new position, then it moves and stays where I dropped it for the session.
2. **(Restore on reopen/reload, item 5, FR-31, AR-6)** Given a moved comment box, when I close and reopen it, or reload the paper (Story 3.5), then it reopens at its persisted position, not snapped back to the pin default.
3. **(Command path + additive contract, item 5, FR-31, AR-6/AR-7, AD-4/AD-8)** Given the persisted position, then it is stored as a **scale-independent CSS-px offset from the pin** (FR-31's finalized wording, PRD `prd-paper-mate-2026-06-28/prd.md` L69 — NOT a normalized page fraction), written through the command path (one undoable zundo step), and added additively to the contract (`Style.bubble_offset_x`/`bubble_offset_y`, no `schema_version` break; a box without them falls back to the pin-relative default, i.e. `{0, 0}`).
4. **(No page-edge clipping, item 5, NFR-1)** Given a box dragged near a page edge, then it still renders in the fixed-overlay path without page-edge clipping (the `comment-bubble-page-edge-clipping` fix already in place) and without reflowing the canvas.
5. **(Click vs. drag, no spurious commits)** Given a plain click on the bubble's empty padding (no real drag), then NO position is committed and NO new undo step is recorded — only a drag past a small pixel slop counts as a move (mirrors the codebase's existing `COMMENT_CLICK_SLOP`/`HANDLE_MOVE_SLOP` convention).
6. **(No regression)** Given the comment bubble's corner-handle resize (`bubble_width`/`bubble_height`, prior feature) and the compact box-comment popup (Story 8.4), then both are unchanged: resize still commits via `resizeCommentAnnotation`, and position persistence composes with a compact bubble's own placement (`rightOf(...)`) the same way it does with a normal pin-anchored bubble.

## Dev Notes

### Unit decision: CSS-px pin-relative offset, NOT a normalized page fraction

The epics.md open design call — *"store the offset in normalized page fractions vs a pin-relative delta"* — is **already resolved by the finalized PRD**: FR-31 (`prd-paper-mate-2026-06-28/prd.md` L69) reads *"stored as a scale-independent offset from the pin"*. Decision: **`bubble_offset_x`/`bubble_offset_y: float | None`, CSS px, scale-independent** — the exact same unit family as `bubble_width`/`bubble_height` (both describe the SAME floating, non-page-anchored popup's own chrome geometry). This is the **opposite** unit choice from Story 10.4's `collapsed_width` (a page-anchored memo box that rides zoom, normalized `[0,1]`) — do not confuse the two. The comment bubble is not part of the page's rendered content; it is chrome that floats near its pin, so a raw CSS-px offset that never changes with zoom is correct and matches how the existing (unpersisted) `dragOffset` local state already behaves today.

### The pin never moves; only the bubble's chrome position gets an additive offset

Resolves the open call *"whether the pin stays fixed while only the box moves (assumed yes)"* — **confirmed yes**. This story never touches `anchor` (the pin's own page-normalized position, AD-4) — only `style.bubble_offset_x/y`. The pin is still the single source of truth for WHERE the comment is anchored on the page; the offset only repositions the floating popup relative to it. `commentScreenPoint()` (`AnnotationInteraction.tsx:306-325`, which computes `pos` from the pin's live/committed anchor) is unchanged; the offset is applied entirely inside `CommentBubble`'s own `transform`, exactly as the existing (currently unpersisted) drag already does.

### Single-id scope, not group-broadcast (resolves the multi-page-group open call)

Resolves *"per-page scope for a multi-page comment group"*. A two-page comment's group siblings (`commentGroupIds`) only ever render as `CommentPreview` (a non-draggable glance popup) on the page where they aren't the `selectedComment` — only ONE page's bubble is ever open and draggable at a time. So `repositionCommentAnnotation` is single-id, mirroring `resizeCommentAnnotation`'s existing scope decision exactly — do NOT route it through `commentGroupIds` the way `retextAnnotation`/`recolorAnnotation` (which mutate shared text/color across the group) do.

### Click-vs-drag slop is an established codebase convention — reuse it, don't invent a new one

Two existing precedents solve exactly this problem elsewhere: `COMMENT_CLICK_SLOP = 5` (`useCreateQuickBox.ts:35`, distinguishes a comment-pin click from a comment-pin drag) and `HANDLE_MOVE_SLOP = 5` (`useEditGesture.ts:61`, distinguishes a plain click on an edit handle from a real move, with a `DragState.moved` boolean gate at `onMove`/`onUp`, `useEditGesture.ts:281-285`/`313-320`). `CommentBubble`'s own position-drag has never had this gate (it's a local, useState-only, non-persisted interaction until now) — without adding it, EVERY click on the bubble's empty padding (which today already sets `boxDragRef` unconditionally on `pointerDown`) would commit a same-or-near-zero-delta write and a spurious undo step once persistence lands. Add `BUBBLE_MOVE_SLOP = 5` (same value, same rationale) and gate the NEW `onReposition` commit on having crossed it — do not gate the visual `dragOffset` state itself (sub-slop jiggle can still update the live transform imperceptibly, matching how `useEditGesture` only gates the COMMIT, not every intermediate preview frame beyond the slop check).

### The page-edge clamp already measures the transform; it just needs to re-run on offset change

The existing `useLayoutEffect` in `CommentBubble.tsx` (L107-119, the Codex-MED page-edge-clamp fix, `comment-bubble-page-edge-clipping`) calls `boxRef.current.getBoundingClientRect()`, which is **transform-aware** — it already reflects the current `dragOffset` translate with zero additional math needed. Its only gap: `dragOffset.x/y` are not in its dependency array, so it doesn't re-fire when the offset changes (mid-drag, or the instant a persisted offset is restored on mount). Since this offset is now a PERSISTED, reload-visible position (not a throwaway transient), add `dragOffset.x, dragOffset.y` to the deps so the clamp reliably re-applies whenever the offset changes, closing this for both the live-drag and on-mount-restore paths (AC #4).

### `CommentPreview` must mirror the offset for hover/selected consistency

`CommentPreview.tsx` already reads `anno.style.bubble_width`/`bubble_height` (L111-112) so its SIZE matches the real bubble even while only hover-previewing (not yet selected). Without also reading the offset, a moved comment's hover-glance preview would visibly snap to the stale pin-relative spot while the actual opened bubble shows the moved position — an inconsistency a user would immediately notice on any moved comment. Task 5 closes this the same way 10.4 closed the analogous memo-preview-size gap: a read-only mirror, no new gesture in `CommentPreview` itself.

### Regressions to guard (AC #6)

- **Resize (`bubble_width`/`bubble_height`, `resizeCommentAnnotation`):** completely untouched — a separate field, a separate store action, a separate UI affordance (the corner handle). Reposition and resize are independent and must both persist simultaneously without one clobbering the other (both live under the same `style` object via spread, so this is automatic as long as neither action's `set()` overwrites the other's field — verify in the store test that a resize-then-reposition, or vice versa, leaves both fields set).
- **Compact box-comment popup (Story 8.4, `isBoxComment`/`compact` prop):** its `pos` is already `rightOf(selectedCommentRawPoint)` and its `transform` is `translate(dragOffset.x, dragOffset.y)` with NO `PIN_OFFSET_TRANSFORM` prefix (L128-130) — the offset composes identically regardless of `compact`, since it's the same `dragOffset` state either way. No branch needed in Task 3; verify live in Task 7(g).
- **Story 10.4 (`collapsed_width`, memo-only):** no interaction — distinct annotation type (`memo` vs `comment`), distinct field, distinct component (`MemoBox` vs `CommentBubble`). Do not touch `MemoBox.tsx`/`useEditGesture.ts`.
- **Text-edit session / Esc / Delete handling inside the bubble:** untouched — this story only adds a slop gate + a new callback to the EXISTING pointer handlers, it does not change `onKeyDown`, focus management, or the textarea/button exclusion logic.
