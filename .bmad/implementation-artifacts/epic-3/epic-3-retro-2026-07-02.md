## What went well

- **AE-2 sequencing held exactly as committed.** Story 5.0 (structural refactor, PR #24, 2026-06-30) merged BEFORE Story 3.1 (command path, PR #25, 2026-07-01) — the one prep item Epic 2's retro flagged as sequencing-critical was honored, and 3.1's Dev Notes confirm it "lands on the clean base Story 5.0 left" instead of adding sprawl for 5.0 to later unwind. This is the direct counter-example to Epic 2's #2 recurring problem (a correct prep item, skipped, that cost a bug + a correct-course) — this time the prep item was actually done first.
- **AE-1 held cleanly.** `zundo` adopted; 3.2 is introduced in its own Dev Notes as "the AE-1 enabler story," wrapping the one command surface so delete/move/recolor/retext all became undoable for free, with the coalescing traps (grouped create, per-keystroke retext, per-pointermove drag) named and handled deliberately (a new batched `addAnnotations` action for the first).
- **A real accidental-data-loss risk caught in PLANNING, not in review.** Story 3.3 explicitly deviated from the `epics.md` AC (Del/Backspace both delete) to Del-ONLY, because Backspace is a high-frequency text-editing key and firing an annotation delete from it is a genuine accidental-deletion risk — written into the story spec itself before a line of code, not found as a bug afterward.
- **Cross-model Codex review (AE-6) caught a real HIGH bug.** Story 3.4 (autosave): a stale in-flight PUT from a previous `docId` could survive a doc switch (the single-flight scheduler's refs were shared, not doc-scoped), letting a late response write the wrong document's snapshot. Noted as "unreachable via the shipped UI today... but a real latent gap" — fixed with a `generationRef`. This is the epic's only labeled HIGH finding, and cross-model review is exactly what caught it (the same practice that caught HIGH bugs in 4 different Epic-2 stories).
- **A live-smoke methodology win, not just a bug catch.** Story 3.7's smoke initially LOOKED like it found a bug (an apparently dropped undo step after clicking "Turn into highlight" from inside a focused textarea), but it turned out to be an artifact of driving the click via an *untrusted* synthetic `dispatchEvent`/`.click()`, which doesn't reproduce a real click's native focus-shift behavior. Re-testing with real trusted CDP input (chrome-devtools MCP `click`/`press_key`) showed the code was correct all along — no fix needed, only a testing-methodology correction. This produced the standing memory `use-trusted-input-for-focus-sensitive-smoke`.
- **The Annotation Bank (3.6) shipped the `EXPERIENCE.md` F2 "climax" scenario** (Ctrl+B → click a highlight → canvas jumps + flashes) as a pure client derivation over the store already hydrated by 3.5 — zero backend/contract touch. Exactly the "foundation pays off, tools become thin" pattern Epic 2's retro named as its #1 win, now repeating in Epic 3.
- **DPR>1 live smoke discipline held consistently across all 8 stories**, by name (`verify-on-hidpi-and-real-host`), at escalating DPR values as the epic went on (1.25 → 1.5 → 2.0) — the most faithfully executed Epic-2 action item (AE-5) of the epic, with one honestly-admitted exception (below).

## Recurring problems / patterns

1. **Artifact/bookkeeping hygiene is now the epic's weak link, in three separate places.** (a) `sprint-status.yaml` lagged real delivery: Stories 3.4/3.5/3.6 merged individually across 2026-07-01 (04:50, 11:38, 18:16) but weren't marked `done` until 2026-07-02 (~02:57), batched — this is *why today's retro kickoff* needed memory + git archaeology just to determine Epic 3's true state (3-8 had already been attempted-and-reverted, but the status file just said `backlog`). (b) Story 3.2's Dev Agent Record is **entirely blank** — no model, no debug log, no completion notes, `Status:` still literally says `review` inside the file — despite every task checkbox being `[x]` and Task 4 requiring a Codex review before done. There's no artifact evidence the review ever ran. (c) `sprint-status.yaml`'s `action_items` list still marks **AE-1 through AE-7 all as `open`**, even though the story text shows most were actively applied or deliberately, reasonably deferred (see AE-4 below) — the tracking file was never updated to reflect real progress. The *narrative* discipline (dev notes, guardrails, live-smoke rigor) stayed consistently strong all epic; the *bookkeeping* discipline had at least three visible lapses.
2. **AE-4 (doc-scope the store) was never actually closed — it was punted through the entire epic.** 3.1: "Full doc-scoping stays AE-4 / 3.4." 3.2: "Do NOT add doc-scoping here." 3.4 — the persistence story itself — still calls it "queued." By 3.5/3.6/3.7 the justification quietly shifts to "there's one doc per page-load in v1 (AD-6, no in-session doc-switch UI)," i.e. the gap was reframed as out-of-scope-by-product-design rather than resolved. This is not free of consequence: it is the EXACT gap behind 3.4's one HIGH Codex finding (a stale PUT surviving a doc switch). The bug was fixed locally (a `generationRef`), but the structural cause (singleton, not doc-scoped, store) is still there for the next feature that touches doc switching to rediscover.
3. **A shared "exemption" helper was reused across two different semantics and broke one of them.** `isExempt()` (skip INPUT/TEXTAREA/SELECT/BUTTON — built so pointer-gesture handlers don't eat a control's click) was reused for the undo/redo keydown handler. Right after creating a mark, focus lands on the quick-box's first swatch `<button>`, and `isExempt` swallowing BUTTON meant `Ctrl+Z` did nothing immediately after a create. Fixed same-day (`855df42`) by giving `useUndoRedo` its own `isEditable()` that exempts only truly-editable fields, not buttons. Same *class* of bug as Epic 1's document-handler lesson: a utility built for one interaction pattern, silently wrong when reused for another with different correct semantics.
4. **The version/lockfile sync miss recurred across epochs.** Story 5.0's own Codex review flagged a stale `server/uv.lock` (still recording the pre-bump version) as its ONLY finding. The identical mistake recurred in this retro's own closeout version bump this session — caught only because `git status` was checked before committing. Two occurrences of the same slip, one epic apart.
5. **AE-7 (Codex-sandbox backend-pytest friction) is still only partially fixed.** The `UV_CACHE_DIR=/tmp/uv-cache` workaround is applied by name in every story's Testing Standards section, but Story 3.1's own dedicated review file reports the backend suite was STILL inconclusive under it ("collected 43 tests but timed out... no failure trace"). No later story reports the failure recurring, so it's unclear whether it was silently resolved or simply not re-triggered.
6. **"Opportunistic fix bundling" is a recurring habit, and one feature shipped entirely outside the story pipeline.** Story 3.6's diff bundles two unrelated user-reported fixes (a comment click-away deselect fix, a Highlight-tool flyout UX change) alongside its own feature. Story 3.1 alone generated nine distinct post-"done" live-smoke fixes (labeled A–I), including one (C) fully reverted and replaced by a different mechanism (E) within the same story. And separately, the movable-comment-box/pin feature (PR #31) landed in the same window as Epic 3's close via an ad-hoc user request (brainstorm → design-spec → plan → implement), never entering `epics.md`'s numbering at all — sound process, but it means `epics.md`'s Epic 3 section doesn't fully describe everything Epic 3 shipped.
7. **One admitted DPR>1 smoke gap.** Story 3.3's live smoke explicitly flags: "cross-page DPR>1.25 grouped-delete path requires manual verification (jsdom/headless cannot test it)" — noted as an outstanding gap, not silently skipped, but never confirmed closed in a later pass. The Epic-2 carryover item (backfill the 2.11 box-select DPR smoke) is similarly still unconfirmed.

## Key takeaways

1. A committed sequencing decision (Story 5.0 before 3.1) held and paid off exactly as predicted — proof that Epic 2's retro discipline (name the risk, commit to an order, honor it) works when actually followed. Repeat this for any future foundation-vs-feature sequencing call.
2. Shared "exemption/guard" utilities need per-call-site semantic review, not blind reuse — the `isExempt`/undo collision is the same class of bug as Epic 1's document-level-handler lesson: a utility correct for one interaction pattern can be silently wrong for another. Audit every NEW call site of a shared guard helper for whether its exemption reasoning still applies.
3. Deferring a piece of debt with a plausible-sounding reason ("out of scope by product design") is not the same as it being safe — AE-4's doc-scoping gap was deferred for a whole epic on reasonable-sounding grounds, then a HIGH bug showed up in exactly that gap anyway. When a deferred item produces a real Codex finding, that is the epic's own signal to stop deferring it.
4. Artifact hygiene (status files, Dev Agent Records, action-item tracking) is now the epic's own weak link, the same way DPR>1 smoke discipline was Epic 2's named gap. Fix the cadence (AE3-1/AE3-2) so the next retro doesn't need memory-file and git archaeology just to establish ground truth.
5. A story can be fully implemented, fully unit-tested, and still be the right call to throw away. Story 3.8's discard (after a rigorous, methodical elimination process ruling out a dozen candidate causes) is not a failure — it is the system working: live smoke at real DPR with real drag input caught a hard external blocker before it shipped, exactly what that discipline exists for.
