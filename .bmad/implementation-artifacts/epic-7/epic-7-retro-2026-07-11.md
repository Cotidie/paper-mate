## What went well

- **Refactor-as-last-story-of-the-epic worked (the epic's cleanest decision).** Resequencing 7-12 to be terminal let one decomposition pass absorb the debt from every feature story at once — 7.9 metadata columns, 7.10 persisted-layout store, 7.11 cell-type/tag seam — instead of five partial refactors. It landed byte-identical-contract (openapi diff = `info.version` only; `schema.d.ts` unchanged), suites green (backend 290, client 1376), and passed an independent cross-model Codex review clean.
- **The byte-identical-contract gate held as the refactor's success test.** A structure-only change proved itself by the contract not moving, not by a new feature — the right discipline for a pure refactor, and it caught nothing because nothing broke.
- **Steering worked.** Wonseok reads the heavy correct-course churn as healthy: the mechanism let the epic adapt to real usage (e.g. removing the author filter once it was felt to fight author editing). The epic ended stronger than the 6-story plan.

## Challenges / patterns

1. **epics.md drifted from shipped reality — a THIRD time (Epic 3 → 6 → 7).** 7-12's ACs were written against a stale snapshot: line counts of 629 vs the actual 950, and an assumption the author *filter* still existed after 7-11 removed it. The story needed a dedicated "Reality corrections" section to compensate. This is the exact gap AE6-4 (and Epic 3 before it) named. The story file was corrected locally; the epic-level source drifted anyway.
2. **Build-then-remove churn inside a story.** The author *filter* was built in 7-11 and removed the same day (2026-07-11, "it makes it hard to edit Authors"). Wonseok judged the overall steering healthy and did not want a gate on it, so this is logged as an observation, not an action.
3. **Two Epic-6 action items recurred because they were never decided, only noted** (see follow-through). This retro decided both.
