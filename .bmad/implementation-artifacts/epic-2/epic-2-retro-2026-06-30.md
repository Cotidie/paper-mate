## What went well

- **The foundation-first bet (Story 2.2) paid off.** 2.2 stood up `anchor/` + the full `Annotation` entity + the Zustand `store/` + the overlay + the quick-box shell, proven on the simplest mark. After that, **almost every tool story reports "implemented client-only, contract byte-identical."** Pen (2.8), memo (2.9), and comment (2.10) all found their contract (`PathAnchor` / `RectAnchor` / `body` / `stroke_width` / `type`) already generated by 2.2, so each tool was a thin feature, not new architecture. This is the Epic-1 PREP-1 investment delivering exactly as bet.
- **Adopt-stable-primitives (AP-4) held.** The anchor layer was built on pdf.js viewport projection + the native Selection API + `Range.getClientRects()`, and the pen on perfect-freehand — never hand-rolled. The one real tension (`convertToPdfPoint` returns y-up, conflicting with the top-left contract) was caught in the 2.2 review and resolved cleanly: `render/` bakes the top-left page box, `anchor/` does pure scale-normalization on top, which round-trips across zoom and `/Rotate`.
- **Cross-model Codex review (AP-3) caught real HIGH bugs the implementer missed, on every story that used it:** disarm-mid-drag persisting a pen stroke and touch/stylus scrolling instead of drawing (2.8), cross-page comment body edits updating only one page sibling (2.10), cross-doc selection leak (2.5), cross-doc mark bleed (2.2). The "review with a different model" practice earned its keep again.
- **DPR>1 live smoke caught geometry bugs jsdom structurally cannot.** The headline is the 2.3 HiDPI bug: the text layer's `--total-scale-factor` wrongly included `devicePixelRatio`, stretching every highlight/selection rect ~DPR× past the glyphs into the page margin — **invisible at DPR=1, caught only at DPR=1.25.** Cross-page leak checks for underline (2.7) and comment (2.10) were verified live at DPR≥1.25. The `[[verify-on-hidpi-and-real-host]]` memory paid off directly.
- **No recurrence of Epic-1's #1 bug class.** Document-level interaction handlers (AP-1) were followed throughout; the focus-bound-handler bug did not return. The render/ mock-barrel sync (AP-2) held every story (every story confirms "no new render export → both barrels untouched").

## Recurring problems (patterns across stories)

1. **The epic was restructured 4 times mid-flight, most of it from one story's live smoke.** Story 2.3's smoke alone spawned the tool FSM (2.4), the selection seam (2.5), and arm-time color (2.6); the original 2.1 bundled five architectural pillars + a feature and had to be split into 2.1/2.2. Healthy that it was caught early and formalized via `correct-course` rather than crammed in, but it signals the **upfront breakdown under-modeled the tool-state machine and the selection seam** — both of which were named Epic-1 prep items.
2. **The orthogonal-state anti-pattern shipped, then had to be unwound.** `mode` (cursor/hand/box) + `armedTool` as two parallel fields let pan stay armed alongside highlight, so the pan handler ate the highlight drag (the 2.3 "no reaction" bug). **Epic-1 PREP-3 predicted exactly this** ("design the overlay state machine once"), but it was skipped before 2.3 — so the bug shipped, got a surgical mutual-exclusion patch, then a full single-`activeTool` FSM story (2.4). The prep item was correct; skipping it cost a bug plus a correct-course.
3. **Cross-doc / cross-page scoping bugs recurred (2.2, 2.5, 2.10).** Same root each time: the singleton store is not doc-scoped yet (persistence is Epic 3), so selection / recolor / comment-group operations can leak across documents. Patched defensively three times. This will keep recurring until Epic 3's ownership model (hydrate-on-open, flush-on-change, AR-6) lands.
4. **Conditional sprawl is now acknowledged debt.** Each tool story adds an arm to the if-chains in `AnnotationInteraction` (create path), `AnnotationLayer` (render groups), the store (`recolor`/`restroke`/`retext`/`resize` twins), and `create.ts` (`build*` twins). Both big files are "too large." Logged as the Epic-5 Story 5.0 refactor (per-kind dispatch table + FSM consolidation + src split).
5. **Two verification gaps.** (a) **Story 2.11 box-select shipped with no DPR>1 live smoke** ("requires a human to run at DPR>1") — a gap against the standing `[[verify-on-hidpi-and-real-host]]` rule. (b) Backend `pytest` could not run in the Codex review sandbox (uv cache on a read-only filesystem + a TestClient hang) on 2.4 and 2.10; the `UV_CACHE_DIR=/tmp/uv-cache` workaround is partial.

## Project Lead input (decisions made in this retro)

- **Command-stack approach (resolves Epic-1 PREP-2): adopt `zundo`.** The Zustand temporal middleware is the adopt-stable (AP-4) choice and fits AR-7 exactly (client-only, in-memory do/undo, discarded on reload). Epic-3 undo/redo (Story 3.2) builds on it rather than a bespoke stack.
- **Sequencing: land the Epic-5 Story 5.0 structural refactor at the Epic-2/Epic-3 boundary, BEFORE Story 3.1.** The deferred-work note and Epic-1 PREP-3 both point here: the dispatch-table + FSM-consolidation + src-split refactor should be the clean base the command path lands on, so 3.1 does not add more sprawl that the refactor then has to unwind. Accepted cost: a short delay to user-facing edit/undo/persist.

## Next epic preview — Epic 3 "Edit, persist & review"

Epic 3 builds the one client command stack (do/undo, AR-7), autosave + the `GET`/`PUT /api/docs/{doc_id}/annotations` routes (the backend grows here — 2.2 defined the entity but not the routes), exact restore on reopen, and the Annotation Bank. **Every Epic-2 edit is currently a client-only direct store mutation with no command stack** (`recolorAnnotation`, `restrokeAnnotation`, `retextAnnotation`, `resizeMemoAnnotation`, `deleteAnnotation`); Story 3.1 must fold all of them onto one command path. The plan is sound; the two carryover decisions above gate its start.

## Significant discovery

**Epic update required: NO.** Epic 3's plan is sound. Two qualifiers, both now resolved into the action items: sequence **Story 5.0 before 3.1**, and adopt **`zundo`** for the command stack (PREP-2). No story rewrite needed.

## Key takeaways

1. The foundation-first story (2.2) is the reason Epic 2's tools were thin — generate the whole entity + the seams once, and the features fall out client-only. Repeat this pattern for Epic 3's command path + persistence seam.
2. A correct prep item skipped is a bug deferred, not avoided — PREP-3 (the overlay FSM) was right, skipping it before 2.3 cost a shipped bug + a correct-course. Do Story 5.0 before 3.1 for the same reason.
3. Cross-model review + DPR>1 live smoke remain the two practices that catch what unit tests cannot — keep both mandatory, and close the 2.11 smoke gap.
