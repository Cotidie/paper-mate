# Story 4.2: Column-aware selection & highlight geometry

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a reader,
I want selection and highlights to respect column boundaries,
so that a drag stays in its column and a same-line highlight never bridges the gutter.

## Context & root cause (read first)

Two-column papers break two things. Both trace to ONE missing concept: the app has **no column model**.

1. **Highlights join across the gutter** (`deferred-work.md` 2026-06-29, user screenshot). A highlight (or any selection-driven mark) over text at the SAME vertical position in the left and right columns paints as ONE band bridging the central gutter, instead of two per-column bands. **Root cause, verified:** `client/src/anchor/index.ts:251` `mergeRects` clusters per-line rects into a band by VERTICAL overlap only (`overlap > 0.5 * min-height`) and unions them, with NO horizontal-contiguity / gutter check. Two runs on the same y-row in different columns get unioned into one rect whose x-extent spans left-column → gutter → right-column. (The cross-page `collectTextRects` fix stops full-PAGE leaks; this same-row union still bridges columns WITHIN one page.)

2. **Multi-column drag-select intrudes the other column** (`deferred-work.md` 2026-06-29, the reverted controller). A drag in one column can pull text from the other because the browser's native Selection is a single contiguous DOM range and the text-layer DOM order interleaves in some regions (title/author header). This was attempted FOUR times and reverted to the clean native baseline at the user's request (`deferred-work.md` L65-82); the codebase was bloating for a feature that needs a design, not more patches.

**These are two DIFFERENT risk tiers. Do not conflate them.** Part A (gutter-split in `mergeRects`) is a small, well-bounded pure-geometry fix and is the high-value core. Part B (column-aware drag-select) is the large, historically failed controller. Sequence and gate them (see "Scope & sequencing" below).

## Scope & sequencing (READ BEFORE CODING — controls what you build)

This story is **design-first with a hard split**. Do them in order; Part B is gated on a design pass and has a defined fallback.

- **Part A — gutter-split (AC 1, AC 4-A): DO THIS FIRST, it is the committed deliverable.** Make same-row rects that are separated by a large horizontal gap (the gutter) NOT union. Pure geometry in `anchor/`, behind the existing `Rect[]` contract, so highlight / underline / preview all inherit it. Low risk, unit-testable, ships value on its own.

- **Part B — column-aware drag-select (AC 2, AC 3, AC 4-B): DESIGN PASS FIRST, then attempt with a fallback.** This is the 4-times-reverted controller. Before writing controller code, produce the written design in the "Part B design" section below (approach, module boundary, how selection/copy/highlight consume it) and **sanity-check it against the two known failure modes** (async `selectionchange` flicker; fighting the browser's contiguous range) AND the [[story-3-8-caretrangefrompoint-blocker]] (mid-drag `caretRangeFromPoint`/`caretPositionFromPoint` corrupt after any prior text-layer interaction in Chromium — any "resolve caret from pointer mid-drag" approach will hit this). **If the design cannot avoid all three, STOP and take the fallback:** ship Part A, write Part B up in `deferred-work.md`, and set the story `blocked` for the Part-B slice (mirror how Story 3.8 was handled) — do NOT start a fifth patch-on-patch attempt. Escalate to Wonseok at that point (see "Questions" at end).

**A story that ships only Part A cleanly is a WIN.** A story that re-bloats the selection code chasing Part B is the exact failure the revert was meant to prevent.

## Acceptance Criteria

1. **(FR-7, FR-13, NFR-3, AR-4/AD-4) Same-line highlight does not bridge the gutter.** Given two text runs on the same visual line in different columns, when a highlight/selection covers one (or both), `mergeRects` does NOT union across a large horizontal gap: each column gets its own band. The fix stays in `anchor/` behind the existing `Rect[]` contract so highlight / underline / preview inherit it. A same-line, single-column selection still merges to one band (no regression to the existing per-line merge).

2. **(FR-7, NFR-3) Drag-select stays within the pointed column.** Given a drag-select inside a two-column body, the selection stays within the column the pointer is in; a cross-column selection is expressed in reading order, column by column (not a gutter-bridging contiguous block). *(Part B — gated per "Scope & sequencing"; if deferred, this AC moves to the follow-up with the story marked `blocked` for this slice.)*

3. **(architecture, AD-9) One column module with a narrow contract.** The column model lives in ONE module with a narrow contract (page text-run geometry → column boundaries / cursor logical position: column + line, or an "else" region for full-width/outside regions) that selection, copy, and highlight create-on-release (`rectsFromSelection`) consume — NOT spread across `render/` + `anchor/` + `annotations/` the way the four reverted attempts did. The projection-profile column detector is the reusable core; the gutter-split in AC 1 consumes the SAME column model (do not build two independent gutter heuristics). *(Part B — gated.)*

4. **Live-smoke at DPR>1 (mandatory — jsdom zeroes rects).**
   - **4-A (Part A):** a cross-column, same-line selection/highlight on a real two-column paper paints as TWO separate per-column bands (no gutter bridge), AND a cross-PAGE selection still splits correctly with no full-page leak, both at DPR>1.
   - **4-B (Part B, if built):** a drag-select inside a two-column body stays in the pointed column, at DPR>1, verified with a REAL trusted mouse drag (not a scripted `Range`), and re-smoked cross-page.

5. **Tests + typecheck + build green.** `cd client && npm test`, `npm run typecheck`, `npm run build` all pass. Any new `render/`/`anchor/` export used by mocked modules is mirrored in BOTH `vi.mock("./render")` barrels (`App.test.tsx`, `Reader.test.tsx`) in the SAME change if it crosses that boundary (Epic-1 retro rule). No new backend change → no OpenAPI/contract regen, no `docs/API.md` change.

With the check in the predicate, the second right fragment skips the left row (gap too big) and correctly matches/extends the right band. ## Part B design

> Filled during Task 2, before any controller code was written.

**Investigation done first.** Before sketching a fifth design, read the actual reverted attempt #4 (`03d471b`, "Fix: column-clipped text selection over the raw pdf.js layer", recoverable via reflog) rather than only the `deferred-work.md` summary. It is the most sophisticated of the 4 attempts and the closest to what this story's AC 2/3 ask for: native selection ran untouched (no fighting the browser), no caret-API calls anywhere (avoided the class of bug that later killed Story 3.8). It failed anyway — per `deferred-work.md:75`, it "reintroduced the native-selection-cancel-on-release regression" (root cause undetermined even after that attempt) AND "clipped legit selections to a single column / whole-column over-selection on blank-space drags" (a correctness bug in the clip-after-the-fact approach itself: it read `window.getSelection()`'s native rects and filtered post-hoc to the anchor's column band, rather than building the span directly from the controller's logical column+line model as the user's direction specifies). This means **the two failure modes named in the Tasks section (flicker, fighting-the-browser) are necessary but not sufficient** — attempt 4 cleared both and still failed on a fresh, unexplained regression.

- **Column detector:** the projection-profile detector from attempt #2/#4 (`deferred-work.md:73`, `03d471b`'s `render/textSelection.ts`) is sound and reusable: for each candidate x, count rows a glyph CROSSES it; a gutter is a central x-strip few lines cross (height-independent, robust to a full-width title and unequal column lengths). This part of the history is NOT in question.

- **What a genuinely new (never-attempted) design would require:** per the user's original direction (`deferred-work.md:77-82`), selection must be built FROM the controller's emitted logical positions (anchor-line → focus-line within a column), not by steering or post-hoc-clipping the browser's native `Range`. Concretely this means abandoning `window.getSelection()` / native `Range` for the mark's geometry entirely: paint a live overlay from precomputed per-line rects during the drag (pure geometry, safe), and at drag-end resolve the exact covered text/offsets via a **manual** `Range.setStart`/`setEnd` + `getClientRects()` binary search over character offsets — deliberately avoiding `caretRangeFromPoint`/`caretPositionFromPoint` altogether.

- **Failure-mode #1 (async `selectionchange` flicker) — avoidable.** Attempts #1-#2 already solved this: pure geometric pointer-vs-line-box comparison, never `elementFromPoint`. Not a live risk for a new design that reuses this piece.

- **Failure-mode #2 (fighting the browser's contiguous range / cancel-on-release) — NOT certifiable as avoided.** Attempt #4 did not touch `Selection.addRange`/`removeAllRanges` and still regressed this. `deferred-work.md` records no resolved root cause for why. A design cannot honestly claim to avoid a regression whose trigger is undetermined even in the most careful prior attempt.

- **Failure-mode #3 ([[story-3-8-caretrangefrompoint-blocker]]) — avoidable ONLY by never calling `caretRangeFromPoint`/`caretPositionFromPoint`, and this is harder than it sounds.** The 3.8 blocker (`deferred-work.md:245`) is explicit: corruption "reproduces 100% deterministically... once ANY prior click or drag has touched the page's text layer" — i.e. every highlight after a reader's FIRST drag on a fresh page load is at risk, not a corner case. AE3-7 (the project's own follow-up item) names "resolve only ONCE on `pointerup`" as an **unvalidated idea for a future revisit**, not a proven-safe pattern — it was never built or tested. A design for THIS story cannot borrow it as a load-bearing assumption. The alternative (manual `Range` + per-character `getClientRects()` binary search, avoiding the caret API family entirely) is plausible but has zero prior evidence it works cleanly over pdf.js's per-span `transform: matrix(...)` glyph substrate — the same substrate the 3.8 investigation flagged as the likely (if unconfirmed) root cause of the caret corruption in the first place. It would need to be prototyped and live-smoked to know, not merely reasoned about on paper.

- **Decision: DEFER.** Two of three failure modes cannot be certified as avoided from the design alone: #2 has an unexplained recurring regression even in the most careful prior attempt, and #3's only escape route (manual per-character `Range` construction) is untested against the exact glyph substrate implicated in a related hard blocker. Per "Scope & sequencing," this is exactly the documented STOP condition: do not start a fifth patch-on-patch attempt. Taking the fallback — ship Part A (Task 1, done), write this analysis to `deferred-work.md`, leave AC 2/3/4-B and Task 3 deferred rather than implemented, and escalate the decision to Wonseok (surfaced in the story's completion communication) rather than unilaterally committing a fifth attempt with no live-smoke validation of the new pieces.
