## Deferred from: story 2-5 blank-space text selection (2026-06-29): multi-column selection controller

> **Reverted to the clean native-selection baseline (`a294ca9`) on 2026-06-29 at the user's request.** Several iterations of a multi-column selection fix were built on top and then dropped because each added more machinery and introduced new regressions; the codebase was getting bloated/spaghettied for a feature that needs a proper design, not more patches. The dropped work is recoverable from git reflog at commit **`03d471b`** ("Fix: column-clipped text selection over the raw pdf.js layer") if any piece is wanted as reference. **Re-evaluated in Story 4.2 (2026-07-02) via a design pass gated on 3 failure modes — deferred again. See "Discarded: Story 4.2 Part B" below for the fifth-attempt analysis.**

**Current (reverted) behavior: the accepted baseline.** Plain browser-native selection over the raw pdf.js text layer (AD-2). For body paragraphs the text-layer DOM is column-sequential, so a drag stays within its column and does NOT intrude the other column. The known limitation we are accepting for now: dragging into a line's trailing blank space does NOT snap the selection to that line's end (it extends to the nearest text in DOM order, so it can over- or under-shoot the pointed line), and an interleaved region (the title/author header, where DOM order alternates columns) can still select across columns. Highlight geometry is already safe via `anchor/collectTextRects` (per-text-node rects, never element boxes, the cross-page fix), so a highlight never fills a page.

**Why the patches failed (history, so we don't repeat them).** Four approaches were tried and dropped:
1. **Line model + `selectionchange` focus correction.** Snapped the focus to the pointed line's end in its own column. Worked for the trailing-gap case but FLICKERED: `selectionchange` is async, so the browser painted the native (wrong-column) focus for a frame before the correction landed.
2. **Geometric (not `elementFromPoint`) gap decision + column-first line model.** Killed the flicker and the blank-below-column jump, detecting columns from a vertical projection profile (a gutter is the central x-strip FEW lines cross; robust to a full-width title and unequal-length columns). But it still steered the browser's single CONTIGUOUS range, which cannot be made column-exclusive when DOM order interleaves.
3. **Own the selection** (suppress native via `selectstart` preventDefault, drive it ourselves in `pointermove` from the column model). Isolated the column but FOUGHT the browser: it provoked `pointercancel` mid-drag and cancelled normal selection on release. Fragile.
4. **Column-clipped overlay** (native runs untouched, hide its paint, render our own clipped overlay). Gave true column isolation in the screenshot, but reintroduced the native-selection-cancel-on-release regression and clipped legit selections to a single column / whole-column over-selection on blank-space drags. Too much surface area.

**The proper approach to design later (user's direction: a LAYERED controller).** Separate "where is the cursor" from "what is selected":
1. A **controller** tracks the cursor's logical position: which COLUMN it is in and which LINE within that column (and an "else" for full-width / outside-any-column regions). Build the column/line model once per drag (the projection-profile column detector plus per-column vertical-center line clustering from approach 2 is the reusable core).
2. The controller **emits** that logical position (column + line, or the else-region) as the pointer moves.
3. **Selection is implemented on top of the controller's emitted positions**, not by steering the browser's contiguous range. This likely means building the selection geometry/paint from the (anchor-line to focus-line within the same column) span directly, rather than relying on `window.getSelection()` to be column-correct.

Plan this thoroughly before coding (own story / correct-course): decide how selection, copy, and the existing highlight create-on-release (`rectsFromSelection`) consume the controller, and how cross-column selection is expressed (e.g. reading-order, column by column). Keep it in ONE module with a narrow contract so it does not spread across `render/`, `anchor/`, and `annotations/` the way the dropped attempts did.

## Deferred from: Story 2.7 underline (2026-06-29) — cross-type recent-wins / unified hit-layer

> Surfaced by the Story 2.7 cross-model code review (Codex, MED). Dismissed for 2.7 with rationale; recorded here as the proper fix to do when multi-type selection is designed.

- **Cross-type recent-wins is not preserved across the two render groups** [`client/src/annotations/AnnotationLayer.tsx`]. Story 2.7 paints highlights in an isolated 0.4-opacity group (`.annotation-highlights`) and underlines in a full-opacity sibling group (`.annotation-underlines`). Because the opacity group is an isolated stacking context, per-mark `z-index` cannot order marks ACROSS the two groups, and the later-in-DOM underline group always stacks above highlights. So for a highlight and underline overlapping the SAME text run, the underline always wins paint + pointer hit-testing regardless of `created_at` — i.e. newest-wins holds WITHIN each type but not BETWEEN types.
  - **Why it's narrow (why 2.7 dismissed it):** only one annotation type existed before 2.7, so cross-type ordering was never specified; within-type recent-wins is intact; the only mis-target is the corner case of a highlight created LATER than an underline on the exact same run (clicking selects the older underline).
  - **Proper fix (do with Epic 3 multi-type selection / Annotation Bank):** separate hit-testing from the visual opacity grouping. Render ONE transparent interaction/hit layer of full-rect divs for ALL marks, ordered by the full `created_at`-sorted `marks` list (so the newest overlapping mark is topmost and wins the pointer), with `pointer-events:auto` + the select/hover handlers; keep the two VISUAL groups (`.annotation-highlights` opacity group + `.annotation-underlines`) as `pointer-events:none` paint-only layers that read `hoveredId`/`selectedId` from the store. This keeps the highlight opacity compositing, the full-opacity underline, and a generous full-rect hit target while restoring global recent-wins. It changes the Story 2.5 selection-seam DOM (handlers/testids move to the hit layer), so it needs its own story + test updates — hence deferred, not patched inside 2.7.

## Bug: copied text loses spaces at line breaks (2026-06-29)

> Reported by the user with a screenshot. Selecting "networks or training small networks directly. This paper proposes a class of network architectures that allows" copies as "networks ortraining small networks directly. This paper proposes aclass of network architectures that allows" — the space at each LINE break is dropped (so the last word of one line fuses with the first word of the next). Bug, not a feature. Lives in the pdf.js text layer (AD-2 raw layer), not the annotation code.

- **Symptom:** copy/clipboard text (and `selection.toString()`) is missing the inter-line whitespace; words that wrap across a line break are concatenated.
- **Root-cause hypothesis** [client/src/render/index.ts `renderPage` → `new TextLayer({...})`]: we build the text layer with pdf.js's `TextLayer` class but do NOT reproduce the bits pdf.js's own web viewer adds around it for correct selection/copy — specifically the per-line whitespace/`<br>` handling and the trailing `endOfContent` element. pdf.js text items carry `hasEOL`; the viewer inserts a line break / whitespace between items so a cross-line copy keeps the space. Our overlay just dumps `offText.childNodes`, so adjacent line spans copy with no separator → fused words. (The intra-line space loss, if any, is the same family: spacing between text items not materialized as text.)
- **Fix path:** reproduce pdf.js's text-layer copy handling — either keep the `endOfContent` element + the viewer's selection CSS, or post-process the text items to emit an EOL whitespace/`<br>` between lines (mirror `TextLayerBuilder` from pdf.js's web viewer). Verify by copying a multi-line selection and diffing against the source text. Keep it in `render/` (the text layer's owner); no annotation/anchor change. Note: highlight/underline GEOMETRY is unaffected (it uses per-line rects), but anything that reads `selection.toString()` for `anchor.text` (the stored highlight text) inherits the same missing spaces, so fixing this also cleans up stored annotation text.

## Bug: highlights on the same line across the two columns join across the gutter (2026-06-29)

> Reported by the user with a screenshot. A highlight (and any selection-driven mark) drawn on text at the same vertical position in the LEFT and RIGHT columns renders as ONE band that bridges the central gutter, instead of two separate per-column bands. Related to — but a distinct root cause from — the reverted "multi-column selection controller" item above.

- **Root-cause hypothesis** [client/src/anchor/index.ts `mergeRects`]: `mergeRects` clusters per-line rects into one band purely by VERTICAL overlap (`>50%` of the smaller height) and unions them, with NO horizontal-contiguity / gutter check. Two text runs on the same visual line in different columns share that y-row, so they get unioned into a single rect whose x-extent spans from the left column across the gutter into the right column → one wide highlight bridging the gap. (The cross-page `collectTextRects` fix prevents full-PAGE leaks, but within ONE page this same-row union still bridges columns.)
- **Fix path:** make `mergeRects` cluster by row AND horizontal contiguity — do not union two rects on the same y-row if a large horizontal gap separates them (e.g. the gutter), or detect column boundaries (the projection-profile column detector noted in the multi-column-controller item is the reusable core). Keep it in `anchor/` (the geometry owner) behind the existing `Rect[]` contract so highlight/underline/preview all inherit the fix. MUST be live-smoked with a cross-column same-line selection at DPR>1. Ties into the deferred multi-column selection controller above (shared column model).

- **Why the failure mode is fatal to the feature as designed:** the core use case — dragging a handle to re-range a mark that was ITSELF just created via a text-selection drag — sits exactly in the failure zone (any prior text-layer interaction poisons subsequent same-region hit-testing during a NEW drag). This is not a corner case to patch around; it is the primary path.
- **If revisited:** needs either (a) a different anchor-recovery strategy that does not call `caretRangeFromPoint`/`caretPositionFromPoint` repeatedly mid-drag — e.g. resolve only ONCE on `pointerup` rather than continuously previewing via native Selection — or (b) confirmation this is a headless/CDP-only artifact via a real headed browser before betting the design on the continuous-preview approach again.

## Discarded: Story 4.2 Part B (column-aware drag-select controller) — design-gated, deferred without a fifth implementation attempt (2026-07-02)

> Story 4.2 ("Column-aware selection & highlight geometry") split into Part A (gutter-split in `mergeRects`, SHIPPED — see the story file / File List) and Part B (a genuine column-aware drag-select controller, AC 2/3/4-B), which the story pre-gated on a written design pass clearing 3 named failure modes before any controller code. This is a **design-only evaluation** — no controller code was written, so nothing was reverted; the story simply did not attempt a fifth implementation.

- **What was newly investigated (beyond the existing summary above):** read the actual reverted attempt #4 (`03d471b`, "Fix: column-clipped text selection over the raw pdf.js layer") rather than relying only on the one-line summary. It is the most sophisticated of the 4 prior attempts and the closest to this story's AC 2/3: native `Selection` ran untouched (didn't fight the browser), and it never called `caretRangeFromPoint`/`caretPositionFromPoint` anywhere. It still failed — per this file's L75 (above): it "reintroduced the native-selection-cancel-on-release regression" (**root cause undetermined**, even in this most-careful attempt) and "clipped legit selections to a single column / whole-column over-selection on blank-space drags" (a correctness bug in the clip-after-the-fact design itself — it read `window.getSelection()`'s native rects and filtered post-hoc to the anchor's column band, rather than building the span directly from a column+line logical model as the user's original direction specified).
- **Why this matters for gating a 5th attempt:** the two failure modes Story 4.2 named up front (async `selectionchange` flicker; fighting the browser's contiguous range) are necessary but **not sufficient** — attempt #4 cleared both and still regressed. A genuinely new design (per the user's original layered-controller direction: build the selection span directly from the controller's anchor-line→focus-line positions, never trusting `window.getSelection()`/native `Range` to be column-correct) would need to abandon `caretRangeFromPoint`/`caretPositionFromPoint` entirely and resolve exact character offsets via manual `Range.setStart`/`setEnd` + `getClientRects()` binary search instead.
- **Why that path was NOT attempted:** two of the three failure modes could not be certified as avoided on paper. (1) The cancel-on-release regression (failure mode #2) has no established root cause even after the most careful prior attempt — a design cannot honestly claim to avoid a trigger nobody has identified. (2) The `caretRangeFromPoint`/`caretPositionFromPoint` corruption (failure mode #3, Story 3.8's blocker) reproduces on EVERY drag after a reader's first on a fresh page load, not a corner case — and AE3-7 (the project's own follow-up item) explicitly names "resolve once on `pointerup`" as an **untested idea for a future revisit**, not a validated escape route. The alternative (manual per-character `Range` construction, avoiding the caret-API family entirely) is plausible but has zero prior evidence against pdf.js's per-span `transform: matrix(...)` glyph substrate — the same substrate the 3.8 investigation flagged (unconfirmed) as the likely root cause of the caret corruption itself. It would need to be prototyped and live-smoked, not reasoned about, and Story 4.2's own scope explicitly asked NOT to start a fifth patch-on-patch attempt without that certainty.
- **Current (unchanged) behavior:** identical to the "Current (reverted) behavior" described in the original multi-column-selection-controller entry above — the clean native-selection baseline, cross-column drag-select still not column-exclusive in the interleaved-DOM (header) case.
- **If revisited:** this needs a dedicated design/prototyping spike (own story, with budget for live-smoke iteration on the manual-Range/binary-search approach against a real two-column paper) BEFORE another full implementation attempt — not a repeat of "design on paper, then build." Also worth an explicit, standalone experiment to nail down failure mode #2's actual trigger (attempt #4 hit it despite not touching `Selection.addRange`/`removeAllRanges`), since an unexplained regression recurring across multiple independent implementation strategies suggests a shared root cause worth isolating on its own before it blocks a 5th attempt too.
