## Deferred from: story 2-5 blank-space text selection (2026-06-29): multi-column selection controller

> **Reverted to the clean native-selection baseline (`a294ca9`) on 2026-06-29 at the user's request.** Several iterations of a multi-column selection fix were built on top and then dropped because each added more machinery and introduced new regressions; the codebase was getting bloated/spaghettied for a feature that needs a proper design, not more patches. The dropped work is recoverable from git reflog at commit **`03d471b`** ("Fix: column-clipped text selection over the raw pdf.js layer") if any piece is wanted as reference. **Re-evaluated in Story 4.2 (2026-07-02) via a design pass gated on 3 failure modes — deferred again. See "Discarded: Story 4.2 Part B" below for the fifth-attempt analysis.**

**Current (reverted) behavior: the accepted baseline.** Plain browser-native selection over the raw pdf.js text layer (AD-2). For body paragraphs the text-layer DOM is column-sequential, so a drag stays within its column and does NOT intrude the other column. The known limitation we are accepting for now: dragging into a line's trailing blank space does NOT snap the selection to that line's end (it extends to the nearest text in DOM order, so it can over- or under-shoot the pointed line), and an interleaved region (the title/author header, where DOM order alternates columns) can still select across columns. Highlight geometry is already safe via `anchor/collectTextRects` (per-text-node rects, never element boxes, the cross-page fix), so a highlight never fills a page.

**Why the patches failed (history, so we don't repeat them).** Four approaches were tried and dropped:
1. **Line model + `selectionchange` focus correction.** Snapped the focus to the pointed line's end in its own column. Worked for the trailing-gap case but FLICKERED: `selectionchange` is async, so the browser painted the native (wrong-column) focus for a frame before the correction landed.
2. **Geometric (not `elementFromPoint`) gap decision + column-first line model.** Killed the flicker and the blank-below-column jump, detecting columns from a vertical projection profile (a gutter is the central x-strip FEW lines cross; robust to a full-width title and unequal-length columns). But it still steered the browser's single CONTIGUOUS range, which cannot be made column-exclusive when DOM order interleaves.
3. **Own the selection** (suppress native via `selectstart` preventDefault, drive it ourselves in `pointermove` from the column model). Isolated the column but FOUGHT the browser: it provoked `pointercancel` mid-drag and cancelled normal selection on release. Fragile.
4. **Column-clipped overlay** (native runs untouched, hide its paint, render our own clipped overlay). Gave true column isolation in the screenshot, but reintroduced the native-selection-cancel-on-release regression and clipped legit selections to a single column / whole-column over-selection on blank-space drags. Too much surface area.

**The proper approach to design later (user's direction: a LAYERED controller).** Separate "where is the cursor" from "what is selected":
1. A **controller** tracks the cursor's logical position: which COLUMN it is in and which LINE within that column (and an "else" for full-width / outside-any-column regions). Build the column/line model once per drag (the projection-profile column detector plus per-column vertical-center line clustering from approach 2 is the reusable core).
2. The controller **emits** that logical position (column + line, or the else-region) as the pointer moves.
3. **Selection is implemented on top of the controller's emitted positions**, not by steering the browser's contiguous range. This likely means building the selection geometry/paint from the (anchor-line to focus-line within the same column) span directly, rather than relying on `window.getSelection()` to be column-correct.

Plan this thoroughly before coding (own story / correct-course): decide how selection, copy, and the existing highlight create-on-release (`rectsFromSelection`) consume the controller, and how cross-column selection is expressed (e.g. reading-order, column by column). Keep it in ONE module with a narrow contract so it does not spread across `render/`, `anchor/`, and `annotations/` the way the dropped attempts did.

## Deferred from: Story 2.7 underline (2026-06-29) — cross-type recent-wins / unified hit-layer

> Surfaced by the Story 2.7 cross-model code review (Codex, MED). Dismissed for 2.7 with rationale; recorded here as the proper fix to do when multi-type selection is designed.

- **Cross-type recent-wins is not preserved across the two render groups** [`client/src/annotations/AnnotationLayer.tsx`]. Story 2.7 paints highlights in an isolated 0.4-opacity group (`.annotation-highlights`) and underlines in a full-opacity sibling group (`.annotation-underlines`). Because the opacity group is an isolated stacking context, per-mark `z-index` cannot order marks ACROSS the two groups, and the later-in-DOM underline group always stacks above highlights. So for a highlight and underline overlapping the SAME text run, the underline always wins paint + pointer hit-testing regardless of `created_at` — i.e. newest-wins holds WITHIN each type but not BETWEEN types.
  - **Why it's narrow (why 2.7 dismissed it):** only one annotation type existed before 2.7, so cross-type ordering was never specified; within-type recent-wins is intact; the only mis-target is the corner case of a highlight created LATER than an underline on the exact same run (clicking selects the older underline).
  - **Proper fix (do with Epic 3 multi-type selection / Annotation Bank):** separate hit-testing from the visual opacity grouping. Render ONE transparent interaction/hit layer of full-rect divs for ALL marks, ordered by the full `created_at`-sorted `marks` list (so the newest overlapping mark is topmost and wins the pointer), with `pointer-events:auto` + the select/hover handlers; keep the two VISUAL groups (`.annotation-highlights` opacity group + `.annotation-underlines`) as `pointer-events:none` paint-only layers that read `hoveredId`/`selectedId` from the store. This keeps the highlight opacity compositing, the full-opacity underline, and a generous full-rect hit target while restoring global recent-wins. It changes the Story 2.5 selection-seam DOM (handlers/testids move to the hit layer), so it needs its own story + test updates — hence deferred, not patched inside 2.7.

## Bug: copied single-visual-line text copies as MULTIPLE clipboard lines (2026-07-07)

> **PROMOTED to Epic 8 Story 8.1 (2026-07-07 correct-course; renumbered from Epic 9 Story 9.1 on 2026-07-11 when Remote sync was un-numbered).** This spec is retained here as the source; the tracked home is the epics.md / sprint-status story. See `sprint-change-proposal-2026-07-07-deferred-review.md` and `sprint-change-proposal-2026-07-11-deprioritize-sync.md`.
>
> Reported by the user with a screenshot: selecting text that reads as one paragraph in the PDF (wrapping across several visual lines) copies with a hard line break at every visual line, not just at real paragraph ends. This is the mirror-image of the copied-text-loses-spaces bug that Story 4.1 fixed (that one lost the space; Story 4.1's fix reproduces pdf.js's `<br>`-per-line behavior, which over-corrects into a hard break where only a soft wrap occurred).

- **Root cause (confirmed by reading the actual pdf.js source, not hypothesized):** `render/index.ts` uses pdf.js's own `TextLayer` class (`node_modules/pdfjs-dist/build/pdf.mjs`), which appends a `<br role="presentation">` after every text item with `hasEOL: true`. `hasEOL` comes straight from `geom.hasEOL` in pdf.js's low-level text-content extraction, which is set from the PDF's own glyph-positioning data - every visually distinct line in the PDF (wrapped or a genuine paragraph end) is a separately Y-positioned glyph run, and a PDF file has NO paragraph-boundary metadata at all once authored. pdf.js therefore cannot distinguish "wrapped for column width" from "real line/paragraph end" from geometry alone, and marks EVERY line `hasEOL: true`. `render/textSelection.ts`'s Story 4.1 fix (above) faithfully reproduces this exact upstream behavior - confirmed this is not a Paper Mate regression; Firefox's own built-in PDF viewer (same pdf.js codebase) has the identical characteristic.
- **Why this is a feature, not a bug fix:** closing it needs a paragraph-vs-wrap HEURISTIC built on top of pdf.js's per-line geometry - e.g. compare consecutive lines' Y-gap against the page's typical line-height, check left-margin/indent alignment, maybe trailing punctuation - and only keep a hard `\n` for a genuine break, joining a wrap with a space instead. Any such heuristic has inherent false positive/negative risk and needs validation against real multi-column, justified, and indented-paragraph PDFs, not just a unit test.
- **Decision (2026-07-07, user):** promoted to **Epic 8 Story 8.1** (paragraph-aware copy; renumbered from Epic 9 Story 9.1 on 2026-07-11). The story starts with a Y-gap/indent heuristic prototype against 2-3 real papers before committing to the full implementation; the join-vs-break rule is settled in that spike.

True in-place text editing means rewriting the PDF's content streams (the glyph-positioning + font operators), which is a heavy, lossy operation:
  - **Reading is fine; writing is the problem.** pdf.js is a renderer, not an editor — it has no API to rewrite text runs. Editing needs a PDF WRITE library (e.g. `pdf-lib`, or `mutool`/`qpdf`/`mupdf` server-side) and, for re-flowing edited text, font subsetting + content-stream surgery. Justified text, ligatures, kerning, and embedded subset fonts (which often lack glyphs you'd type) make faithful edits very hard; results commonly shift layout or fall back to a different font.
  - **Style-only changes (bold/italic/color) are also non-trivial:** bold/italic usually require a DIFFERENT font program (not a flag), so you must embed a matching face and re-lay the run; color is the most tractable (a fill-color operator change) but still a content-stream edit.
- **Two architectural routes to weigh in the spike:**
  1. **Annotation-overlay "edit" (no PDF mutation, fits the current architecture):** cover the original run with a white-ish box and draw replacement text / styled text on the overlay (a memo/textbox-like mark, Story 2.9 family). Keeps the source PDF immutable (AD-6/AD-8 ownership), stays client-side, no new heavy dep — but it is a visual overlay, not a real PDF edit (copy/extraction still yields the original text, and export would need to flatten the overlay).
  2. **Real PDF content editing (new capability, backend-grows):** introduce a server-side PDF write/edit library and an export path; this breaks the "backend is a dumb store" posture (AD-6), needs font handling, and is realistically a Phase-2/3 (export) or post-v1 effort. The persisted-format/MAJOR-version implications (AD-8) apply if edits are saved into the stored PDF.
- **Recommendation:** do a dedicated feasibility spike (decide overlay-edit vs. true-edit, pick the library, prototype color-only first as the tractable slice) BEFORE writing a story. Likely lands as: color/style-via-overlay in the annotation epic, true text editing deferred to a Phase-2/3 "edit & export" thread. Couples with the already-deferred export-with-highlights (Phase 2) since both need a PDF write path.

## Discarded: story 3-8-adjust-text-range (2026-07-02) — hard Chromium browser blocker, descoped from v1 (FR-26)

> Attempted in full on branch `story-3-8-adjust-text-range` (anchor `caretRangeAtPoint`, a `rerangeTextGroup` reconciler, `store.reanchorTextGroup`, `AnnotationLayer` start/end drag handles, `useTextRangeGesture`) — all unit tests passed, but live smoke (own scratch dev servers, DPR=2, Chromium 149) hit a hard, deterministic browser bug during an in-flight drag. Branch deleted, all changes reverted; `main` carries no trace. Epic-3 retro (2026-07-02) descoped this from v1: `epic-3` closes without it, FR-26 stays open for a future revisit.

- **The bug:** during an active native mouse-button-held drag (`pointerdown` → `pointermove`s → `pointerup`), both `document.caretRangeFromPoint(x, y)` and `document.caretPositionFromPoint(x, y)` return wrong/empty results (empty `<span>`, offset 0) for screen points that resolve CORRECTLY when queried standalone (outside a drag) at the identical coordinates. Worst case: the reconciler resolved the wrong page entirely (jumped 1-2 pages away), producing garbage annotations.
- **Reproduces 100% deterministically** at fixed screen coordinates once ANY prior click or drag has touched the page's text layer (a fresh reload with zero prior interaction does NOT trigger it at that same region — the FIRST drag on a fresh load is clean, every drag after is poisoned).
- **Eliminated by direct A/B test** (none of these were the cause): devicePixelRatio (reproduces at DPR=1 too), `setPointerCapture`, `e.preventDefault()` placement/timing, `selection.addRange()` mid-drag, `user-select:none`, `selectstart` prevention, window/container scroll (none occurred), our own overlay DOM (`.annotation-highlight`/handles — removing them entirely from the DOM did not fix it), a React re-render race (waiting 150ms for a suppression class to commit did not fix it), double-click/selection-granularity carryover (a 2s gap before the second drag did not fix it).
- **Not specific to this app's code:** a raw `document.addEventListener('pointermove', ...)` probe with zero app logic showed the identical corruption. Most likely a Chromium hit-testing quirk interacting with pdf.js's per-span `transform: matrix(...)` glyph-width correction, since that's the most unusual thing about the text substrate versus a normal prose page. Root cause otherwise undetermined; not confirmed whether this is a headless/CDP-only artifact (no headed-browser test was possible in that environment).
- **Why the failure mode is fatal to the feature as designed:** the core use case — dragging a handle to re-range a mark that was ITSELF just created via a text-selection drag — sits exactly in the failure zone (any prior text-layer interaction poisons subsequent same-region hit-testing during a NEW drag). This is not a corner case to patch around; it is the primary path.
- **If revisited:** needs either (a) a different anchor-recovery strategy that does not call `caretRangeFromPoint`/`caretPositionFromPoint` repeatedly mid-drag — e.g. resolve only ONCE on `pointerup` rather than continuously previewing via native Selection — or (b) confirmation this is a headless/CDP-only artifact via a real headed browser before betting the design on the continuous-preview approach again.

## Discarded: Story 8.9 (snap empty-space drag to nearest text) — both named techniques failed live smoke, Story 8.8's no-op stays shipped (2026-07-12)

> Story 8.9 was explicitly scoped as a spike: prototype technique (A) then, only if it failed, technique (B), live-smoke each against a real two-column paper (`fixtures/sample-pdfs/Multi-task self-supervised visual learning.pdf`, DPR=1.25, own dev servers, Chromium via the `claude-in-chrome` extension) across REPEATED same-session drags before writing production code. Both techniques failed, for two DIFFERENT reasons. No production code lands; all spike changes (a temporary probe in `render/textSelection.ts`, a prototype `render/nearestTextAnchor.ts`) were reverted (`git checkout -- client/src/render/textSelection.ts`, `rm client/src/render/nearestTextAnchor.ts`) — `main`/the story branch carries no trace beyond this write-up.

- **Technique (A) — `caretRangeFromPoint` resolved once at pointerdown:** the AC-2 decisive A/B measurement (fixed screen coordinates, compare a fresh-load first empty-space drag vs a repeated same-session drag after prior text-layer interaction) reproduced the EXACT corruption signature `deferred-work.md`'s Story 3.8 entry (above) documented: empty resolved node, offset 0. Out of 10 repeated calls at the identical fixed coordinates (interleaved with on-text drags/clicks elsewhere on the page to "touch" the text layer), 3 returned the corrupted empty/offset-0 result and 7 resolved correctly — non-deterministic, but a genuine, repeated recurrence, not a one-off. This confirms AE3-7's untested "resolve once, not continuously" idea does NOT dodge the poison even called exactly once at pointerdown, before any pointermove: the poison is not specific to CONTINUOUS mid-drag calls (Story 3.8's finding) — a single pre-move call over previously-touched text-layer state is enough to trigger it some of the time. Per AC 2's own stated bar ("Do NOT try to patch around it"), technique (A) was abandoned immediately once this reproduced; no root-cause investigation or mitigation was attempted (matches the existing elimination list's conclusion that there is no known patch).
- **Technique (B) — manual `Range` + `getClientRects()` binary search:** implemented in full as a prototype (`render/nearestTextAnchor.ts`: `groupSpanLines` line-band grouping by vertical span overlap, `nearestLine` with preceding-line tiebreak, `nearestSpanInLine` horizontal nearest-span, `nearestOffsetInTextNode` binary-searching character boundaries via non-collapsed single-character sub-ranges — mirrors `anchor/index.ts#collectTextRects`'s per-text-node measurement pattern, all rect readers injectable). Two things were found:
  - **A real bug, fixed during the spike:** pdf.js marks a rotated glyph run (e.g. the margin-printed arXiv identifier, rotated -90deg via a `--rotate` CSS custom property) with a POST-transform `getBoundingClientRect()` that is axis-aligned and page-tall. Naively including it in `groupSpanLines` let ONE rotated span's giant vertical band absorb-merge every normal line whose Y-range it overlapped (nearly the whole page) into one giant "line," so `nearestSpanInLine`'s horizontal step picked an unrelated line's span. Fix: skip any span where `getComputedStyle(span).getPropertyValue("--rotate")` is non-empty/non-zero before line-grouping. After the fix, the resolver correctly and CONSISTENTLY (5/5 repeated same-session calls, no flakiness, unlike technique A) resolved the pointer position (640,655) in the fixture's page-1 right-margin blank space to the correct nearest character — the end of "...NYU depth prediction." (offset 45 = node length, correctly clamped since the pointer x was past the line's last glyph).
  - **A fatal, un-patchable finding:** AC 3's core mechanism — "resolve once + `selection.collapse(node, offset)` + let native drag-extension run" — does NOT work when the pointerdown's DOWN PIXEL itself misses actual glyph content (lands on the bare `.textLayer` div background, which is exactly the empty-origin case this story exists to handle). Confirmed via a direct control: a drag starting 1px INSIDE the last glyph of the same line (x=599 instead of x=640, ~40px into the blank margin) correctly armed native drag-selection and extended through the following paragraphs; the identical drag from the blank-space point (x=640) stayed fully collapsed (`rangeCount:1, isCollapsed:true, text:""`) regardless of `selection.collapse()` to the correctly-resolved anchor, and regardless of whether `event.preventDefault()` was called on `pointerdown`, on `selectstart`, on both, or on neither. The browser's native mouse-selection "arm" state machine requires ITS OWN hit-test at the mousedown pixel to succeed before it will track subsequent `pointermove`s as a selection-extend gesture; a script-level `Selection` mutation after the fact cannot retroactively arm it. This is a browser behavior, not an app bug — no combination of event timing/prevention available to page script was found to work around it.
- **Why this is genuinely un-patchable within scope, not a 5th attempt waiting to happen:** making technique (B) actually extend the selection would require driving the selection ourselves on every `pointermove` (tracking the drag's current point, computing the current line/offset, and calling `selection.extend(node, offset)` continuously) — i.e., reopening exactly the continuous column-aware drag-select controller this story's Dev Notes named as the hard scope guard ("do NOT drive the selection yourself in `pointermove` from a logical column/line model. Every one of those regressed and was reverted"). The story's own budget explicitly forbids that trade for this spike.
- **Current (unchanged) behavior:** Story 8.8's no-op — an empty-space-origin drag does nothing, exactly as shipped. No regression: the reverted spike touched no shipped file.
- **If revisited:** the reusable finding is technique (B)'s anchor-resolution HALF (`groupSpanLines`/`nearestLine`/`nearestSpanInLine`/`nearestOffsetInTextNode`, including the `--rotate` filter) — it is correct and non-flaky standalone. What's missing is a continuous, `pointermove`-driven extension from that anchor that stays within a single column (the exact problem the four originally-reverted multi-column attempts and Story 4.2 Part B were gated on). A future attempt would need to either (a) accept building bounded `pointermove`-driven extension as an explicit, budgeted design (not a spike extension) with the failure modes Story 4.2 Part B's design-gate named, or (b) find a way to make the browser's native arm-on-mousedown succeed at an off-glyph pixel (unresearched; may not exist as a scriptable capability at all).
