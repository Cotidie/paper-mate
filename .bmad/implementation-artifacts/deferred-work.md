## Deferred from: story 2-5 blank-space text selection (2026-06-29): multi-column selection controller

> **Reverted to the clean native-selection baseline (`a294ca9`) on 2026-06-29 at the user's request.** Several iterations of a multi-column selection fix were built on top and then dropped because each added more machinery and introduced new regressions; the codebase was getting bloated/spaghettied for a feature that needs a proper design, not more patches. The dropped work is recoverable from git reflog at commit **`03d471b`** ("Fix: column-clipped text selection over the raw pdf.js layer") if any piece is wanted as reference.

**Current (reverted) behavior: the accepted baseline.** Plain browser-native selection over the raw pdf.js text layer (AD-2). For body paragraphs the text-layer DOM is column-sequential, so a drag stays within its column and does NOT intrude the other column. The known limitation we are accepting for now: dragging into a line's trailing blank space does NOT snap the selection to that line's end (it extends to the nearest text in DOM order, so it can over- or under-shoot the pointed line), and an interleaved region (the title/author header, where DOM order alternates columns) can still select across columns. Highlight geometry is already safe via `anchor/collectTextRects` (per-text-node rects, never element boxes, the cross-page fix), so a highlight never fills a page.

**Why the patches failed (history, so we don't repeat them).** Four approaches were tried and dropped:
1. **Line model + `selectionchange` focus correction.** Snapped the focus to the pointed line's end in its own column. Worked for the trailing-gap case but FLICKERED: `selectionchange` is async, so the browser painted the native (wrong-column) focus for a frame before the correction landed.
2. **Geometric (not `elementFromPoint`) gap decision + column-first line model.** Killed the flicker and the blank-below-column jump, detecting columns from a vertical projection profile (a gutter is the central x-strip FEW lines cross; robust to a full-width title and unequal-length columns). But it still steered the browser's single CONTIGUOUS range, which cannot be made column-exclusive when DOM order interleaves.
3. **Own the selection** (suppress native via `selectstart` preventDefault, drive it ourselves in `pointermove` from the column model). Isolated the column but FOUGHT the browser: it provoked `pointercancel` mid-drag and cancelled normal selection on release. Fragile.
4. **Column-clipped overlay** (native runs untouched, hide its paint, render our own clipped overlay). Gave true column isolation in the screenshot, but reintroduced the native-selection-cancel-on-release regression and clipped legit selections to a single column / whole-column over-selection on blank-space drags. Too much surface area.

**The proper approach to design later (user's direction: a LAYERED controller).** Separate "where is the cursor" from "what is selected":
1. A **controller** tracks the cursor's logical position: which COLUMN it is in and which LINE within that column (and an "else" for full-width / outside-any-column regions). Build the column/line model once per drag (the projection-profile column detector plus per-column vertical-center line clustering from approach 2 is the reusable core).
2. The controller **emits** that logical position (column + line, or the else-region) as the pointer moves.
3. **Selection is implemented on top of the controller's emitted positions**, not by steering the browser's contiguous range. This likely means building the selection geometry/paint from the (anchor-line to focus-line within the same column) span directly, rather than relying on `window.getSelection()` to be column-correct.

Plan this thoroughly before coding (own story / correct-course): decide how selection, copy, and the existing highlight create-on-release (`rectsFromSelection`) consume the controller, and how cross-column selection is expressed (e.g. reading-order, column by column). Keep it in ONE module with a narrow contract so it does not spread across `render/`, `anchor/`, and `annotations/` the way the dropped attempts did.

## Deferred from: Story 2.7 underline (2026-06-29) — cross-type recent-wins / unified hit-layer

> Surfaced by the Story 2.7 cross-model code review (Codex, MED). Dismissed for 2.7 with rationale; recorded here as the proper fix to do when multi-type selection is designed.

- **Cross-type recent-wins is not preserved across the two render groups** [`client/src/annotations/AnnotationLayer.tsx`]. Story 2.7 paints highlights in an isolated 0.4-opacity group (`.annotation-highlights`) and underlines in a full-opacity sibling group (`.annotation-underlines`). Because the opacity group is an isolated stacking context, per-mark `z-index` cannot order marks ACROSS the two groups, and the later-in-DOM underline group always stacks above highlights. So for a highlight and underline overlapping the SAME text run, the underline always wins paint + pointer hit-testing regardless of `created_at` — i.e. newest-wins holds WITHIN each type but not BETWEEN types.
  - **Why it's narrow (why 2.7 dismissed it):** only one annotation type existed before 2.7, so cross-type ordering was never specified; within-type recent-wins is intact; the only mis-target is the corner case of a highlight created LATER than an underline on the exact same run (clicking selects the older underline).
  - **Proper fix (do with Epic 3 multi-type selection / Annotation Bank):** separate hit-testing from the visual opacity grouping. Render ONE transparent interaction/hit layer of full-rect divs for ALL marks, ordered by the full `created_at`-sorted `marks` list (so the newest overlapping mark is topmost and wins the pointer), with `pointer-events:auto` + the select/hover handlers; keep the two VISUAL groups (`.annotation-highlights` opacity group + `.annotation-underlines`) as `pointer-events:none` paint-only layers that read `hoveredId`/`selectedId` from the store. This keeps the highlight opacity compositing, the full-opacity underline, and a generous full-rect hit target while restoring global recent-wins. It changes the Story 2.5 selection-seam DOM (handlers/testids move to the hit layer), so it needs its own story + test updates — hence deferred, not patched inside 2.7.

## Bug: copied text loses spaces at line breaks (2026-06-29)

> Reported by the user with a screenshot. Selecting "networks or training small networks directly. This paper proposes a class of network architectures that allows" copies as "networks ortraining small networks directly. ## Feature request (needs feasibility spike): directly edit selected PDF text + its style (2026-06-29)

> Requested by the user: as a reader, select text in the PDF and edit it OR change its style (bold / italic / color) in place. Logged with a feasibility note because it is a large, architecture-touching capability, NOT a simple annotation — needs a spike before any story.

- **Feasibility (short answer: hard, and a different class of feature than v1).** v1 renders PDFs read-only via pdf.js (AD-2) and layers annotations OVER the page; it never mutates PDF content. True in-place text editing means rewriting the PDF's content streams (the glyph-positioning + font operators), which is a heavy, lossy operation:
  - **Reading is fine; writing is the problem.** pdf.js is a renderer, not an editor — it has no API to rewrite text runs. Editing needs a PDF WRITE library (e.g. `pdf-lib`, or `mutool`/`qpdf`/`mupdf` server-side) and, for re-flowing edited text, font subsetting + content-stream surgery. Justified text, ligatures, kerning, and embedded subset fonts (which often lack glyphs you'd type) make faithful edits very hard; results commonly shift layout or fall back to a different font.
  - **Style-only changes (bold/italic/color) are also non-trivial:** bold/italic usually require a DIFFERENT font program (not a flag), so you must embed a matching face and re-lay the run; color is the most tractable (a fill-color operator change) but still a content-stream edit.
- **Two architectural routes to weigh in the spike:**
  1. **Annotation-overlay "edit" (no PDF mutation, fits the current architecture):** cover the original run with a white-ish box and draw replacement text / styled text on the overlay (a memo/textbox-like mark, Story 2.9 family). Keeps the source PDF immutable (AD-6/AD-8 ownership), stays client-side, no new heavy dep — but it is a visual overlay, not a real PDF edit (copy/extraction still yields the original text, and export would need to flatten the overlay).
  2. **Real PDF content editing (new capability, backend-grows):** introduce a server-side PDF write/edit library and an export path; this breaks the "backend is a dumb store" posture (AD-6), needs font handling, and is realistically a Phase-2/3 (export) or post-v1 effort. The persisted-format/MAJOR-version implications (AD-8) apply if edits are saved into the stored PDF.
- **Recommendation:** do a dedicated feasibility spike (decide overlay-edit vs. true-edit, pick the library, prototype color-only first as the tractable slice) BEFORE writing a story. Likely lands as: color/style-via-overlay in the annotation epic, true text editing deferred to a Phase-2/3 "edit & export" thread. Couples with the already-deferred export-with-highlights (Phase 2) since both need a PDF write path.
