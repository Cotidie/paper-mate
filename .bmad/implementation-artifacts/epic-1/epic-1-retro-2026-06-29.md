## Recurring problems (patterns across stories)

1. **Focus-bound keyboard/pointer handlers — the #1 bug class.** The same defect recurred in **three** stories: 1-5 (zoom keys, HIGH), 1-8 (hold-Space, Medium), 1-9 (focus return after jump, Medium). Root cause every time: a handler bound to `.pdf-canvas` instead of the document level, so it was bypassed or lost when focus moved. We re-learned it three times.
2. **Transient-overlay state machines.** 1-9 shipped "this PDF has no table of contents" when it actually meant "outline still loading" (`[]` conflated with pending). The quick-box (Epic 2) is the same overlay-state shape and will hit the same class of bug.
3. **Render-mock barrel drift.** Every new `render/` export (`pageWindow`/`WINDOW_RADIUS` in 1.7, `panScroll` in 1.8, `getOutline` in 1.9) broke *all* Reader/App tests until the `vi.mock("./render")` barrels were hand-synced. Repeated, predictable footgun.
4. **Backend error-envelope contract.** 1-1 / 1-2 / 1-3 each had review findings where errors escaped the `{ detail: string }` envelope (storage errors as 500s, 422 validation shape, OpenAPI media/error declaration). Settled now, but cost three rounds to converge.
5. **jsdom can't prove interactions.** Every interaction story (scroll, zoom, pan, jump, no-reflow) needed a live Playwright browser smoke for the real AC — jsdom only asserts wiring, not movement/layout. The live smoke became the de-facto AC verifier.

## Project Lead input

- **Standing principle: don't reinvent wheels.** When a stable, proven solution exists, actively adopt it over a from-scratch build. (Saved as durable guidance.) Directly reframes the Epic 2 `anchor/` decision below: use stable primitives under the custom overlay, don't hand-roll the math. Caveat: this does not override deliberate architectural choices like AD-2 (raw pdf.js + custom overlay) — that overlay is an intentional UX decision; apply the principle to the primitives beneath it.

## Epic 2 ("Annotate") preview & readiness

Epic 2 marks up the page with six tools via drag-to-annotate + the contextual quick-box, proves the spatial-anchor model holds across zoom (NFR-3), and defines the `Annotation` entity (AR-5). 7 stories (2-1 … 2-7).

**The big risk — the anchor layer is net-new.** Epic 1 deliberately built **zero** screen↔PDF coordinate math (`render/` only, page box reserved, no `anchor/`). Story 2-1 stands up the entire `anchor/` layer from scratch (normalize against the rendered page box, canonical `{x0,y0,x1,y1}` top-left origin, two-page `group_id` split) **and** introduces Zustand + the command stack (both avoided through all of Epic 1). This is the architectural through-line — get it right once; all of Phase 2/3 consumes it.

**Apply the don't-reinvent principle here.** The `anchor/` math should adopt stable primitives, not hand-roll:
- pdf.js `viewport.convertToPdfPoint` / `convertToViewportPoint` for screen↔PDF conversion (the AD-4 page box is the viewport at scale 1.0 — these methods already do the projection).
- the native Selection API + `Range.getClientRects()` for text-run → rects (the pdf.js text layer already lays positioned spans; let the browser do selection).
- the already-chosen perfect-freehand for the pen (good — a stable lib, not from-scratch).
- a known command-stack pattern (e.g. zundo / immer patches) for undo/redo (Epic 3) rather than a bespoke stack.

**Lessons that apply directly to Epic 2:** the focus-handler convention (Epic 2 adds many armed-tool states + the quick-box overlay) and the transient-overlay state-machine design (distinct pending/empty/armed states + focus return).
