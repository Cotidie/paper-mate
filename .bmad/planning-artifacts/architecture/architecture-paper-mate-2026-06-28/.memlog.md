- (constraint) VERIFIED 2026-06: Firefox has NO native File System Access for local disk (showDirectoryPicker), only OPFS sandbox. Pure-web cannot reach ~/.paper-mate on Firefox. Forces a local backend or desktop shell to satisfy 'Chrome+Firefox + on-disk folder' (PRD).
- (version) PDF.js (pdfjs-dist) = standard web PDF renderer; layered canvas/text/annotation. Custom annotation overlay to be built ourselves; pdf.js built-in annotation layer is for embedded form/link annots only.
- (decision) AD runtime: Option A. SPA (Chrome/Firefox at localhost) + local backend that owns disk I/O. Backend runs as a DOCKERIZED container. ~/.paper-mate reached via host volume mount into the container. Binds: client never touches disk directly; all persistence goes through backend API. Prevents: split per-browser storage behavior + Firefox disk gap.
- (question) DEFERRED (Phase 3 boundary): dockerized backend cannot exec HOST agent CLIs (Claude/Codex/Antigravity). Resolve at Phase 3: host bridge / mount docker socket / agents installed in image / sidecar. Agent abstraction must not assume same-process exec.
- (decision) AD stack: Backend = Python/FastAPI (user preference) in Docker, Pydantic v2 models. Frontend = React 19 + Vite 7 SPA + TypeScript (NOT Next.js: localhost canvas tool, no SSR/SEO, separate Python backend makes Next's server redundant). Renderer = pdfjs-dist raw with custom overlay layers.
- (decision) AD contract sync: annotation model + API contract owned by backend Pydantic models -> FastAPI auto OpenAPI -> generated TS types for client. Binds: client types are generated, never hand-authored, to keep Python<->TS in sync. Prevents: drift between server schema and client model.
- (decision) AD repo: monorepo with client/ (Vite SPA), server/ (FastAPI), and a generated client-types artifact from OpenAPI. Single-user, REST/JSON over localhost, no realtime.
- (decision) AD spatial-anchor (CORE invariant): all annotation geometry stored in page-NORMALIZED coords (fractions [0,1] of page width/height, top-left origin), never screen px or raw PDF points. Screen position is always derived = frac x current_page_pixel_size. Anchor = {page_index, normalized geometry, optional text}. Text anchor = list of normalized quad rects + text snapshot string. Geometric anchor = normalized rect (memo/box) or normalized point list (pen). Binds: every phase (annotations, Phase2 ref-previews, Phase3 click-to-chat targeting) speaks this one anchor vocabulary. Prevents: zoom/DPI/resize drift (NFR-3) + incompatible per-feature coordinate schemes.
- (decision) AD annotation model: one Annotation entity {id, doc_id, type(highlight|underline|pen|memo|comment), anchor, style(color,stroke_width?), body(text|null), created_at, updated_at}; a doc's annotations = flat collection keyed by doc_id.
- (decision) AD ownership: backend filesystem (per-doc annotations file) is durable source of truth; client store is a working copy hydrated on open, flushed on change. Single user, one session per doc, no concurrency.
- (decision) AD state-mutation (invariant): every annotation change flows through ONE path = client command stack (do/undo) -> store -> dirty -> debounced autosave. Undo/redo is client-only in-memory, discarded on reload. Backend is a DUMB store: GET returns last-saved set, PUT overwrites with full current set (whole-document granularity), atomic write (temp+rename). No server-side history/undo/edit-logic. Binds: no component mutates annotations directly; backend never interprets edits. Prevents: divergent edit paths, broken undo, lost data (NFR-4).
- (decision) SEED: client annotation store = Zustand (light, fits single-store working copy). Swappable; not load-bearing.
- (decision) AD storage layout: ~/.paper-mate/library/{doc_id}/ holds source.pdf + annotations.json + meta.json (filename,title,page_count,added,last_opened); ~/.paper-mate/config.json reserved for Phase3 vendor settings. Host folder volume-mounted into container. doc_id = SHA-256 of original PDF bytes, computed ONCE at import, stored, never recomputed. Annotations stored separately, never written into source.pdf, so doc_id is stable across annotating. Binds: same PDF re-added -> same doc_id -> same folder -> existing annotations found (natural dedup). Prevents: annotation loss on re-import, split folders per identical paper.
- (decision) AD frontend modules: Render layer (pdfjs-dist wrapper: page canvas+text layer, exposes page pixel size + projection primitives, knows nothing of annotations) / Anchor service (SINGLE home of normalized-anchor<->screen math) / Annotation layer (renders store annotations, projects via anchor service, pure view) / Tool system (armed-tool state, drag capture, quick-box; selection->anchor service->command->store) / Store (Zustand working copy + command stack) / API client (generated from OpenAPI; hydrate + debounced PUT).
- (decision) AD backend modules: API layer (FastAPI routes: import PDF, list/get docs, serve PDF bytes, GET/PUT annotations) / Storage module (ONLY code touching ~/.paper-mate: hashing, atomic write, layout) / Agent boundary (one reserved interface, Phase 3, not built).
- (decision) AD boundary invariants: (1) anchor math exists ONLY in the anchor service; render layer never knows annotations, tool/annotation features never compute screen<->PDF coords themselves. (2) storage module is the ONLY disk writer; routes never touch fs directly. (3) client reaches backend ONLY via the generated API client. Prevents: incompatible coordinate conversions across features (adversary divergence), scattered disk writes, hand-rolled drifting API calls.
- (decision) AD deployment: single container via docker-compose. FastAPI/uvicorn serves BOTH the API and the built Vite SPA static assets (same-origin -> no CORS). docker-compose.yml volume-mounts host ~/.paper-mate -> container /data, maps port; host path + port via env. Dev: Vite dev server HMR proxies /api to FastAPI. Prod: FastAPI serves dist/. No auth (localhost single-user). No CI/cloud/scaling (out of scope).
